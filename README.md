# SSR Latency Benchmark: Ziex vs Leptos

Head-to-head SSR latency comparison of two full-stack frameworks, each rendering
the same server-side data list, benchmarked with [`oha`](https://github.com/hatoo/oha).
Everything runs in Docker so the result is reproducible.

- **Ziex** (Zig, `0.1.0-dev`, Zig 0.16.0) - `zig build serve`, file-routed page at `app/pages/page.zx`
- **Leptos** (Rust, `0.8`) - Actix-web server rendering a Leptos component to an HTML string per request

Both serve on container port `3000`. Both render the identical workload used by
Ziex's own published benchmark: a `<main>` containing 50 `<div>SSR {v}-{i}</div>`
rows generated by a server-side loop, wrapped in a full HTML document. The list is
produced fresh on every request through each framework's real SSR rendering path
(Ziex's `{for (...)}` template loop; Leptos's `view!` + `.collect_view()`).

## Layout

```
ssr-bench/
├── ziex-app/        Ziex app (starter template reduced to one SSR page) + Dockerfile
├── leptos-app/      Axum + Leptos SSR server + Dockerfile
├── docker-compose.yml   ziex -> localhost:3001, leptos -> localhost:3002
├── bench.sh         builds, starts, waits, runs oha, prints comparison
└── results/         raw oha JSON output
```

## Run

```bash
./bench.sh
```

Tunables (env vars):

```bash
DURATION=30s CONNECTIONS=100 ./bench.sh
```

The script:
1. `docker compose up -d --build` (builds both images, starts both servers)
2. Waits until each server answers on its host port
3. Runs `oha` from a container on the shared docker network, hitting each server
   directly (`http://ziex:3000/`, `http://leptos:3000/`) to avoid host port-mapping noise
4. Parses the JSON and prints a req/sec + p50/p90/p99 latency table

Stop the servers:

```bash
docker compose down
```

## Notes on fairness

- Both render the same 50-row list per request with no caching, generated by a
  real server-side loop through each framework's SSR engine (not a cached or
  static string). Ziex uses its production `serve` path (`--release=fast`);
  Leptos is built `--release` with LTO.
- One asymmetry remains: this Ziex dev version always compiles a client WASM/JS
  bundle and injects a `<link rel=preload>` + `<script>` into the `<head>` (its
  hydration runtime), even with `config.csr = false`. Those are two fixed extra
  tags in the response; `oha` does not fetch the referenced sub-resources, so
  they add a constant number of bytes but no extra render work. Leptos here ships
  pure SSR with no client bundle (its output does include Leptos's `<!>` text
  hydration markers). So if anything Ziex emits slightly more markup per request.
- Load is generated from a separate container on the same bridge network, so both
  frameworks face identical network conditions.
- Numbers depend on host hardware and Docker CPU allocation. Treat them as a
  relative comparison on one machine, not absolute throughput claims.

## Results

Host: Apple Silicon (arm64), Docker Desktop. Load: `oha`, 20s, 50 connections,
requests issued container-to-container over the shared bridge network. Workload:
50-row server-side list render per request. Raw data in `results/*.json`.

| metric       | Ziex (Zig) | Leptos (Rust) |
| ------------ | ---------: | ------------: |
| req/sec      |   ~87,900  |      ~157,100 |
| avg latency  |    0.57 ms |       0.32 ms |
| p50          |    0.44 ms |       0.24 ms |
| p90          |    1.13 ms |       0.46 ms |
| p99          |    2.50 ms |       1.95 ms |
| max          |   14.14 ms |      19.30 ms |
| success      |       100% |          100% |
| total (20s)  |     ~1.76M |        ~3.14M |

**Takeaways**

- With real list-render work (50 rows/request), **Leptos was faster on this
  machine**: ~79% higher throughput and lower latency at p50/p90/p99.
- The tail converges: p99 is close (~1.95 vs ~2.50 ms) and the single slowest
  request actually went to Leptos (19.3 vs 14.1 ms), so the max is noise.
- Both remain fast in absolute terms - sub-millisecond p50, millions of requests
  over 20s, 100% success.
- This does **not** reproduce Ziex's published benchmark (which shows Ziex ahead
  of Leptos). Likely differences: their harness, hardware, and the fact that
  their Leptos target is the full `cargo-leptos` hydration build. Ziex is pre-1.0
  (`0.1.0-dev`), so its `serve` path is not final.

To draw stronger conclusions, raise the workload (e.g. `CONNECTIONS=200`, a larger
list) and run on the target deployment hardware rather than a laptop.
